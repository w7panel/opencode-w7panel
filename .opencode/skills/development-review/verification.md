# 验证整合

## 铁律

```
NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE
```

声称完成而未验证 = 欺诈，非效率。

## 概述

**核心原则**：证据优先，始终如此。

违反此规则 = 违反验证精神。

## 调用场景

development-review 检查通过后，必须经过本子技能验证才能声称"完成"。

## 调用流程

```
1. IDENTIFY: 什么命令能证明这个说法？
2. RUN: 执行完整命令
3. READ: 完整输出，检查退出码，统计失败数
4. VERIFY: 输出是否证实了说法？
   - 否：基于证据说明实际状态
   - 是：基于证据说明结果
5. ONLY THEN: 做出声明

跳过任何步骤 = 欺骗，非验证
```

## 常见场景

| 声称 | 需要 | 不充分 |
|------|------|--------|
| 测试通过 | 测试命令输出：0 failures | 上次运行、"应该通过" |
| Lint 干净 | Lint 输出：0 errors | 部分检查、推断 |
| 构建成功 | 构建命令：exit 0 | Lint 通过、日志看起来不错 |
| Bug 已修复 | 测试原始症状：通过 | 代码改了、假设已修复 |
| 回归测试有效 | 红绿循环验证 | 测试通过一次 |

## 禁止行为

**出现以下想法时，停止并验证：**

- 使用"应该"、"可能"、"似乎"
- 验证前表达满意（"完美！"、"完成！"等）
- 提交/推送/PR 前未验证
- 相信 agent 的成功报告
- 依赖部分验证
- 认为"就这一次"
- 累了想赶紧结束
- **任何暗示成功但未运行验证的措辞**

## 正确模式

**测试**：
```
✅ [运行测试命令] [看到: 34/34 通过] "所有测试通过"
❌ "应该通过了" / "看起来正确"
```

**构建**：
```
✅ [运行构建] [看到: exit 0] "构建通过"
❌ "Lint 通过了" (Lint 不检查编译)
```

**声称**：
```
✅ "构建通过，exit code 0"
❌ "应该没问题了"
```

## 验证模板

```
运行: [完整命令]
预期: [预期结果]
实际: [实际结果]
状态: ✅ 通过 / ❌ 失败
```

## 为什么重要

- 未经验证的声称会导致返工
- 信任一旦失去很难恢复
- 证据优先 = 诚实核心价值
