# 调试整合

## 铁律

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

未完成根因调查 = 不能修复。

## 概述

随机修复浪费时间并制造新Bug。快速补丁掩盖根本问题。

**核心原则**：始终先找到根因再尝试修复。症状修复 = 失败。

## 何时使用

- 测试失败
- 生产Bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别使用场景：**
- 时间紧迫时（紧急情况容易猜测）
- "快速修复一下"看起来很明显时
- 已经尝试过多个修复
- 上次修复无效
- 不完全理解问题

## 四阶段流程

### Phase 1: 根因调查

**尝试任何修复前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 通常包含确切解决方案
   - 完全阅读堆栈跟踪

2. **一致地复现**
   - 能可靠地触发吗？
   - 确切步骤是什么？
   - 每次都发生吗？
   - 如果不可复现 → 收集更多数据，不要猜测

3. **检查最近变更**
   - 什么变更可能导致这个？
   - Git diff、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **多组件系统收集证据**
   - 在每个组件边界添加诊断

### Phase 2: 模式分析

**修复前找到模式：**

1. **找到工作示例**
   - 在相同代码库中找到类似的工作代码
   - 什么类似的东西是有效的？

2. **对比参考**
   - 如果实现模式，完全阅读参考实现
   - 不要跳读 - 每行都读

3. **识别差异**
   - 工作和坏的区别是什么？
   - 列出每个差异，不管多小
   - 不要假设"那不重要"

4. **理解依赖**
   - 这个还需要什么其他组件？
   - 什么设置、配置、环境？
   - 它做什么假设？

### Phase 3: 假设与测试

**科学方法：**

1. **形成单一假设**
   - 清楚陈述："我认为X是根因，因为Y"
   - 写下来

2. **最小化测试**
   - 做最小可能改变来测试假设
   - 一次一个变量

3. **验证后继续**
   - 有效？→ Phase 4
   - 无效？→ 形成新假设
   - 不要在上面加更多修复

### Phase 4: 实现

**修复根因，不是症状：**

1. **创建失败测试用例**
   - 最简单的复现
   - 自动化测试（如果可能）
   - 修复前必须有

2. **实现单一修复**
   - 解决已识别的根因
   - 一次一个改变
   - 不要"顺便"改进

3. **验证修复**
   - 测试现在通过？
   - 其他测试没坏？
   - 问题实际解决了？

4. **如果修复不工作**
   - 停止
   - 数：你尝试了多少次修复？
   - 如果 < 3：返回 Phase 1，用新信息重新分析
   - **如果 ≥ 3：停止并质疑架构**

## 禁止行为

**出现以下想法时，停止并回到 Phase 1：**
- "快速修复一下，以后再调查"
- "先试试改X看看是否有效"
- 添加多个改变，运行测试"
- "跳过测试，我会手动验证"
- "可能是X，让我修复那个"
- "我不完全理解但这可能有效"
- "模式说X但我会适应它"
- **"再试一次修复"（已经尝试2+次后）**

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|---------|---------|
| **1. 根因** | 读错误、复现、检查变更 | 理解什么和为什么 |
| **2. 模式** | 找到工作示例、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小测试 | 确认或新假设 |
| **4. 实现** | 创建测试、修复、验证 | Bug解决、测试通过 |
