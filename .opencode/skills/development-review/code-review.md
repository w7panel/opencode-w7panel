# 代码审查整合

## 概述

**核心原则**：早审查，常审查。

## 何时请求审查

**必需：**
- 完成主要功能后
- 合并到主分支前

**可选但有价值：**
- 卡住时（新视角）
- 重构前（基线检查）
- 复杂Bug修复后

## 调用场景

development-review + verification 通过后，提交前调用。

## 调用流程

```
1. 推送代码
2. 创建 PR
3. 等待人工审查
```

## 审查清单

### 功能完整性
- [ ] 实现了所有需求？
- [ ] 边缘情况处理？
- [ ] 错误处理？

### 代码质量
- [ ] 代码清晰易读？
- [ ] 命名一致？
- [ ] 无冗余代码？

### 测试
- [ ] 有测试覆盖？
- [ ] 测试有意义（不是mock）？
- [ ] 边缘情况测试？

### 规范
- [ ] 遵循项目规范？
- [ ] 无敏感信息泄露？
- [ ] 文档更新？

## 禁止行为

**永远不要：**
- 因为"简单"跳过审查
- 忽略关键问题
- 进行中有未修复的重要问题
- 与有效的技术反馈争辩

## 处理反馈

- 立即修复关键问题
- 修复重要问题后再继续
- 记录小问题以便后续
- 如果审查者错了，反驳（带理由）

## 正确模式

**审查前：**
```bash
# 获取 git SHAs
BASE_SHA=$(git rev-parse HEAD~1)
HEAD_SHA=$(git rev-parse HEAD)
```

**审查后：**
```
审查结果：
├── 优点: [列出优点]
├── 问题:
│   ├── 关键: [必须修复]
│   ├── 重要: [修复后再继续]
│   └── 小: [后续处理]
└── 评估: [是否可以继续]
```

## 与其他子技能的关系

```
开发完成 → development-review
              │
              ├── 失败 → debugging.md
              │
              ├── 通过 → verification.md
              │             │
              │             └── 验证通过
              │
              └── code-review.md → 人工审查 → 提交
```
